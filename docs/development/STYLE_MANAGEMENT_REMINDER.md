# 样式管理开发提醒

## 重要提醒

每次修改页面样式前，必须先确认本项目现有的样式体系是否已经能满足需求，避免重复定义、局部硬写和样式层级混乱。

## 开发前检查清单

- [ ] 是否已查看 `docs/development/DEVELOPMENT_GUIDELINES.md`
- [ ] 是否已查看 `docs/development/STYLE_ARCHITECTURE.md`
- [ ] 是否已检查 `assets/styles/` 中是否已有可复用的 token 或全局变量
- [ ] 是否已检查 `assets/css/` 中是否已有对应页面或模块的统一样式文件
- [ ] 是否已检查当前页面、相似页面、相关组件里是否已有可直接复用的 class
- [ ] 是否已判断这次改动属于 token、统一样式文件，还是组件内局部样式

## 新增样式时的固定顺序

### 先复用

先检查以下内容是否已经存在：

- 现成的 CSS 变量，例如 `var(--color-primary)`、`var(--spacing-md)`
- 现成的统一样式类，例如页面级、模块级、组件级 class
- 已存在的页面专属样式文件，例如 `assets/css/blog.css`
- 相似页面已经实现过的视觉写法

如果已有可用方案，直接调用，不重复新增。

### 缺了先定义

如果现有样式不够表达需求，不要先在页面里硬写，再事后抽离。正确顺序是先补定义：

1. 设计值缺失：
   先补到 `assets/styles/` 的 token / 变量层
2. 可复用页面或模块样式缺失：
   先补到对应的 `assets/css/*.css`
3. 只在当前组件内使用：
   再补到组件自己的 `<style scoped>`

### 最后调用

定义补完后，再在具体页面或组件中：

- 通过 `var(--xxx)` 调用 CSS 变量
- 通过 class 调用统一样式
- 仅在必须动态计算时使用 `:style`

一句话规则：

**先复用，缺了先定义，再在业务文件里调用；有现成的不新增，没有再补齐。**

## 禁止事项

- 禁止在 template 中写固定内联样式
- 禁止先在页面里硬编码颜色、圆角、阴影、间距，再事后回收
- 禁止发现已有相同样式后仍重复新增一套
- 禁止把本应提取到 token 或统一 CSS 文件的样式长期留在页面局部

## 样式落点判断

### 放到 `assets/styles/`

适用于：

- 颜色、圆角、阴影、间距、字号等设计 token
- 跨页面共享的基础变量

### 放到 `assets/css/`

适用于：

- 页面级统一样式
- 模块级统一样式
- 未来大概率会被多个组件或多个区块复用的样式

### 放到 `<style scoped>`

适用于：

- 当前组件特有结构
- 明确不会被其他页面复用的局部样式

### 放到 `:style`

仅适用于：

- 运行时动态位置
- 运行时动态颜色
- 运行时动态动画参数

## 推荐工作流

1. 先找现有变量、样式类、页面样式文件
2. 确认不能复用后，再补样式定义
3. 定义完成后，在页面或组件中接入
4. 自检是否存在“本可复用却重复新增”的情况
5. 自检是否有“本应抽离却仍硬写在页面里”的情况

## 自检问题

每次改样式前后，至少问自己这几个问题：

1. 这个样式能不能直接复用现有 token 或 class？
2. 如果不能复用，是不是应该先补统一定义，再调用？
3. 这个样式到底属于全局、页面级，还是组件私有？
4. 有没有把本应抽离的样式直接写进 template 或局部逻辑里？

## 记住这个原则

样式在本项目里不是“哪里要效果就在哪里写”，而是：

**先找现成样式体系，缺失时先补样式定义，最后在具体文件中调用。**
